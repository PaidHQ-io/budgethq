## Backlog

1. **Spreadsheet-style redesign of Budget, Tagger, and Reporting & Pacing tabs** — make these views look/feel more like Google Sheets or Excel (grid lines, cell-based interaction, familiar spreadsheet affordances) rather than the current card/table styling.
2. **Finish "export to Google Sheets / Excel Online"** — one-shot "create a new file" export shipped for Google Sheets (Dashboard, Tagger, Budget, Pacing → "···" menu). Remaining: Excel Online (Microsoft Graph API) equivalent export.
3. **Append to an existing Google Sheet / Excel file** — current export always creates a brand-new file. Add a mode to pick an existing spreadsheet and append/update rows in it instead (for recurring reporting into the same tracker file).
4. **LinkedIn bulk export enrichment tool** — Mo has the underlying data/spec from another chat; needs to be pulled in when this is picked up.
5. **Inline field/column creation** — add new columns directly within the table UI, spreadsheet-style, instead of only through the existing dimension-management UI.
6. **Inline formulas** — support Sheets/Excel-style formulas on computed columns/cells.
7. **Proactive alerts (Phase 3)** — Slack + email notifications on budget pacing issues. Originally picked as the most exciting next feature (before the multi-tenant backend work took priority); `budgethq.alert_rules` table already scaffolded in the schema, nothing built on top of it yet. Needs Vercel Cron + Slack/email delivery.
8. **Enterprise SSO (SAML/OIDC)** — for enterprise clients whose IT requires it. Needs a paid Supabase plan (not available on Free tier) and per-client configuration against their specific identity provider (Okta, Azure AD, etc.). Deliberately not built speculatively — build when a real enterprise client asks for it.
9. **Pricing tiers, billing, and checkout (Stripe)** — required to actually sell BudgetHQ/PaidHQ access. `core.entitlements` table already supports plan/status per workspace-product, currently only settable manually via API (used today just to grant free trials on signup). Needs: Stripe product/price setup, checkout flow, webhook handler to keep entitlements in sync with subscription status, and a billing/upgrade UI.
10. **Public API + webhooks** — since we can't integrate *into* the pipeline/performance tools orgs already use (Dreamdata, HockeyStack, PowerBI, Salesforce — no API access to those), the better move is integrating *out*: expose BudgetHQ's own spend/budget/pacing data via a public API and webhooks so customers can pull it into whatever BI/pipeline tool they already run. Solves the same problem from the other direction without needing anyone else's API.
11. **Native Google Ads + Meta API sync** — LinkedIn, Bing, and Capterra already pull live via API; Google and Meta are still CSV-only. Native OAuth-based sync for these two would kill the single biggest remaining manual-upload friction point in the core product. **Priority bumped 2026-07-19** — aubado's Budget Checker already has live read-only sync to Google Ads and Meta today (see competitive note below); this is the clearest place BudgetHQ is currently behind, not just "would be nice."
12. **Client-facing shareable views** — a clean, branded, read-only link agencies can share with a client showing pacing/budget status, without giving them full app access. Different from export — a live view, not a static file.
13. **Approval workflows** — budget increase/reallocation requests that require a manager's sign-off, with an audit trail. Enterprise-oriented, complements SSO/billing.
14. **Granular permissions** — beyond owner/admin/member, roles like "can tag but not see budget dollar amounts" — useful for agencies with junior staff or client stakeholders in a workspace.
15. **Budget templates/cloning** — clone last year's budget structure into a new year, or a template into a new client workspace. Saves real setup time for agencies onboarding clients.
16. **Audit log** — a full "who changed what budget when" change feed, distinct from version history (which is snapshot/undo, not a per-change log). Feeds enterprise compliance asks alongside SSO/SOC 2 down the line.
17. **Upload/import history** — a log of past imports (CSV/XLSX, screenshots, Google Sheets connects) showing what was imported, when, and into which segments/tags. Right now nothing is retained once an import modal closes — no way to look back and see where a given row of data actually came from. Distinct from Version History, which snapshots the resulting budget state, not the import events themselves.
18. **View connected/synced Google Sheets** — a place to see which Google Sheets have been pulled from via the manual Connect flow (Budget + Tagger), so re-pulling the latest numbers doesn't mean re-finding and re-pasting the URL every time. Natural companion to the persistent live-sync work already in progress — even ahead of that landing, remembering recently-connected sheets for one-click re-pull is useful on its own.
19. **View connected/synced Excel Online sheets** — same idea as above, once Excel Online connect itself exists (see item 2 — only Google Sheets export/connect are built today; Excel Online has neither yet).
20. **Visualization (graphs and charts)** — everything today is tables/grids (Budget, Tagger, Reporting & Pacing). Add actual charts — spend-over-time, pacing-vs-plan, breakdowns by segment/channel/region — probably starting on the Dashboard and Reporting & Pacing views where a visual read matters most.
21. **AI summary for the Budget tab** — a written summary of the current budget structure (e.g. total allocated, biggest segments, notable gaps like unbudgeted or untagged segments) using the same Ask AI/analyze infrastructure already built.
22. **AI summary for the Reporting & Pacing tab** — a written summary of current pacing (ahead/behind plan, which segments are driving it, anything that needs attention), same underlying AI infrastructure as the Budget tab summary above.
23. **Pre-built PDF/other-format reports** — ready-made report templates (e.g. "Monthly Pacing Summary," "Budget Overview") combining tables, visualization (charts/graphs, see item 20), and an optional AI summary (see items 21–22) into one polished exportable document, instead of the current flat per-view CSV/XLSX/PDF/HTML exports.
24. **Custom PDF/other-format reports** — same idea as above but user-built: pick which tables, charts, and sections (with optional AI summary) go into the report, rather than a fixed template. Natural pairing with item 23 — pre-built templates as a starting point, custom as the flexible option on top.
25. **Automated budget pacing (write-back)** — instead of just tracking/warning on pacing (today's Reporting & Pacing tab), actually push daily/periodic budget adjustments to the live ad platforms so campaigns pace evenly on their own. A real step up from item 11 (Native Google Ads + Meta API sync), which is read-only — this needs write access (OAuth scopes that can modify budgets, not just read spend) plus a rules engine for how/when to adjust. Biggest scope jump of anything on this list; worth its own design pass before starting.
26. **Overspend protection (enforcement)** — auto-pause or auto-cap a campaign when it's about to exceed its budgeted amount, not just warn about it. Complements item 7 (Proactive alerts, which is notify-only) and depends on the same write-back API access as item 25 — the two are natural to build together.
27. **Collaboration: notes & mentions** — leave a note on a budget row/segment and @-tag a teammate to take action, visible to everyone in the workspace. Distinct from item 13 (Approval workflows, which is sign-off/audit-trail) and item 14 (Granular permissions, which is access control) — this is lightweight in-app communication tied to specific budget rows, closer to a comment thread than a governance feature.
28. **Custom budget cycle lengths/frequencies** — today budgets are fixed to calendar months/quarters/years (the `MONTHS`/`QUARTERS` structure). Some agencies run non-calendar cycles — 4-week sprints, bi-weekly, a client's custom fiscal calendar. Would need a real data-model change, not just a UI option — bigger than it sounds.
29. **AI-driven budget recommendations** — proactive, data-driven suggestions for budget changes (not just "here's your pacing," but "increase Google by $2k next month based on X") plus predicted performance impact of a proposed change. Related to the "Forecasting / scenario modeling" item below but different in kind: forecasting is the user modeling their own what-ifs, this is the system actively recommending moves. Depends on having enough historical performance data to make recommendations worth trusting — likely sequenced after visualization (item 20) and AI summaries (items 21–22).
30. **Spreadsheet-style grid editing** — keyboard navigation (arrow keys/tab/enter between cells), multi-cell copy-paste (including pasting a range copied straight from an actual spreadsheet), fill-down, and frozen header row/first column on the Budget grid and Reporting & Pacing tables. Deliberately scoped to interaction behavior only, not a visual reskin — BudgetHQ's dimension-based segments, warning badges, rollup cards, and not-budgeted flags are the actual differentiation over a plain spreadsheet and should stay. Pairs naturally with the Google Sheets connect/export already built.
31. **Persistent/live Google Sheets sync** — the actual phase-two of the manual Connect work: server-side OAuth authorization-code flow + stored refresh token + a sync schedule, so a connected sheet refreshes on its own instead of requiring a manual re-pull. Explicitly agreed as the follow-up to this session's manual-connect build, but wasn't itemized on its own until now — items 18/19 assumed it existed as a tracked line.
32. **Excel Online connect (import)** — companion to item 2 (which only covers Excel Online *export*): read an Excel Online file via Microsoft Graph API into the same header-row-picker/format-detection pipeline Google Sheets connect uses today.
33. **Multi-currency support** — budgets and rollups currently assume a single currency. Needed for agencies with international clients — per-segment currency, plus FX conversion for cross-currency rollup totals.
34. **Bulk actions on the budget grid** — select multiple rows and apply an action across all of them at once: bulk % increase, bulk mark-not-budgeted, bulk delete. Natural complement to item 30's spreadsheet-style editing, but a distinct feature (multi-row selection + action, not cell-level editing).
35. **In-app notification center** — a notification bell/inbox inside BudgetHQ for pacing issues and other alerts, distinct from item 7 (Slack/email delivery) — for whoever's actively in the app but not watching Slack/email at that moment.
36. **White-labeling for client-facing shareable views** — agency's own logo/branding (and ideally a custom domain) on the read-only client links from item 12, instead of visibly being a third-party tool the agency is using.
37. **Onboarding / guided setup flow** — a first-run wizard (connect first ad account, set up first budget) instead of dropping a new workspace into an empty app. Matters more once self-serve billing/signup (item 9) exists and users aren't all hand-held through setup.
38. **SOC 2 / security compliance** — formalizing this as its own item rather than the passing mention under item 16 (Audit log); typically a hard requirement to close real enterprise deals alongside SSO (item 8), not just a nice-to-have.

## Longer-term / known but not scheduled

- Building out VaultHQ, ReportingHQ, and AuditHQ themselves on top of the paidhq-core foundation (currently only BudgetHQ is a real product on the platform).
- **Mobile app** (native or PWA) for on-the-go pacing checks.
- **Anomaly detection** — statistical/ML flagging of unusual spend patterns (spikes, CPC/CPM anomalies), beyond the simple threshold-based pacing alerts in item 7.
- **Forecasting / scenario modeling** — "what if we increase Google spend 20% next quarter" — budget scenario planning beyond the current pacing projections. See item 29 for the related-but-distinct "system recommends" version of this.

## Competitive positioning notes

Running log of what Mo finds while testing/evaluating other tools — useful later for messaging, a pricing/positioning page, or just to keep straight what BudgetHQ is actually competing with (and isn't).

- **2026-07-19 — Optmyzr:** Solid for PPC bid/performance management and automation (rules, scripts, alerts), but weak-to-nonexistent for budget management specifically — no way to set budgets or track spend against custom segments, only account- and campaign-level. Makes sense given what it's built for: bid optimization operates on campaigns/ad groups that already exist, using the platform's own units, not a business's own dimensions (Product, Region, Funnel, etc.). This is true of most platform-native and PPC-automation tools generally — they optimize spend within a structure, they don't help plan/allocate a budget across a business's own segments before spend happens, or track pacing against that plan. Worth remembering as a clean differentiator: BudgetHQ isn't competing with bid-management tools, it's solving a problem most of them don't touch at all.
- **2026-07-19 — competitor feature-page screenshot (budget-pacing tool):** six headline features — Automated Budget Pacing, Overspend Protection, Budget Management, Super-Charged Collaboration, Ultimate Flexibility, Budget Recommendations. Ran each against BudgetHQ: "Budget Management" (custom budgets + spend alerts) and most of "Ultimate Flexibility" (multi-platform tracking, future targets) are already core BudgetHQ strengths. The other four exposed real gaps, now items 25–28 above: automated write-back pacing and enforcement (we currently only monitor/warn, we don't push changes to ad platforms), notes/@-mentions collaboration (we have multi-user workspaces but no in-app commenting), and custom (non-calendar) budget cycle lengths. "Budget Recommendations" maps to item 29, distinguished from the existing forecasting/scenario-modeling idea below. Useful signal: this competitor is positioned around *automation and enforcement* on top of a budget, where BudgetHQ today is stronger on the planning/tracking/segmentation side — the write-back items (25–26) are the biggest lever to close that gap if it matters competitively.
- **2026-07-19 — aubado (aubado.com):** genuinely the closest business-model match found so far — not just a competing budget tool, but the same structural bet as PaidHQ: a modular family of single-purpose marketing apps sold à la carte (~€9.59/app/month, or a bundled "Ensemble" plan), workspace-based with paid add-on workspaces for extra clients/brands. Directly validates the multi-product/pay-per-app PaidHQ model (BudgetHQ, VaultHQ, ReportingHQ, AuditHQ). Their "Budget Checker" app (the direct BudgetHQ competitor) reads as monitoring-only — real-time spend-vs-budget across Google Ads/Meta/LinkedIn — with no sign of custom business dimensions, budget planning/import, or rollups; same platform-native-units gap as Optmyzr above. What they do have today that BudgetHQ doesn't: live read-only OAuth sync to Google Ads and Meta (item 11, bumped to higher priority because of this), and a separate AI co-pilot product ("Duetto," weekly structured account reviews — spend shifts, scaling opportunities, waste) conceptually close to items 21, 22, and 29 but shipped as its own standalone product rather than a feature inside the budget tool. Also pre-launch (waitlist, "founding member" pricing) — an early-stage validation of the same wedge, not an entrenched incumbent. Worth revisiting their pricing model (per-app, per-workspace) when item 9 (Stripe/billing) gets built.
